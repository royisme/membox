# Agent Engineering Rules

Read this file before adding dependencies, tests, public APIs, or repository workflow changes.

## Dependency Policy

Runtime dependencies should remain small and aligned with `pyproject.toml`:

- Required: `pydantic`, `typer`, `rich`.
- Optional LLM: `openai` only behind injectable extractor/embedder implementations.
- Optional AST: `tree-sitter`, `tree-sitter-python` only for source analysis.
- Development: `pytest`, `pytest-cov`, `ruff`, `mypy`, `pre-commit`.

Do not add heavy runtime dependencies for storage, vector search, queues, web servers, or ORMs unless the spec is explicitly revised.

## Testing Expectations

Tests must not depend on external LLM or embedding APIs. Use fake extractor/embedder implementations.

High-value test areas from the spec:

- Entity disambiguation: exact string, case normalization, alias matching, embedding synonym merge, and negative non-merge cases.
- Relation deduplication: unique triples plus multiple evidence documents.
- Predicate normalization: `developed` / `develop` / `开发` → `develops`.
- Multi-hop retrieval: 2-hop and 3-hop BFS recall, plus unrelated entity exclusion.
- Context assembly: returned prompt includes full path and source evidence.
- Provenance: relation-to-document evidence lookup.
- Concurrency: multi-thread writes do not duplicate same entity and final counts are correct.
- SQLite constraints: foreign keys are enabled and enforced.

Prefer red-green-refactor for new domain behavior: add a focused failing test, implement the smallest passing logic, then refactor.

## Validation Commands

Prefer `uv run` because the project declares dependency groups. If `uv` is unavailable, use the corresponding `python -m ...` command in an activated environment.

```bash
uv run ruff check src tests
uv run ruff format --check src tests
uv run mypy src tests
uv run pytest
uv run pytest --cov=membox
uv run membox --help
```

Before finishing a code change, run the narrowest relevant test first, then broader validation when appropriate. Do not claim validation passed unless the command was actually run and succeeded.

## Implementation Style

- Python target is `3.13`; use modern typing syntax such as `str | None` and built-in generics.
- Keep `mypy` strict compatibility: avoid untyped public functions and ambiguous `Any` leaks.
- Keep Ruff compatibility: line length `100`, double quotes, import sorting, no stray `print` in library code.
- Every functional Python file under `src/`, `scripts/`, and `tests/` must start with a module docstring or leading `#` comment explaining its purpose.
- The repository map generator reads these headers and fails if a functional Python file is missing one.
- Prefer small pure functions for normalization, graph traversal, and prompt assembly.
- Keep public API stable and typed from `membox.__init__` once exposed.
- Comments should explain non-obvious constraints or trade-offs, not restate code.

## Release Automation

Use stdlib-only scripts for release bookkeeping:

```bash
uv run python scripts/bump_version.py x.y.z
uv run python scripts/bump_version.py x.y.z --check
uv run python scripts/generate_changelog.py --version x.y.z
uv run python scripts/generate_changelog.py --check
```

Rules:

- Do not manually edit version fields in `pyproject.toml` or `src/membox/__init__.py` during release prep.
- Generate `CHANGELOG.md` from conventional commit history; review generated entries before committing.
- Keep release automation network-free and dependency-free unless the workflow is intentionally revised.

## Repository Map Maintenance

`docs/repository-map.md` is generated by `scripts/update_repository_map.py` and should be kept current by automation. It includes file purpose summaries extracted from module docstrings or leading comments.

Run this after creating, moving, or deleting files:

```bash
uv run python scripts/update_repository_map.py
```

The pre-commit hook also runs this script. If the hook updates `docs/repository-map.md`, review and stage the generated file before committing.
